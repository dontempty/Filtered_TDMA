# Filtered_TDMA 진행상황 종합 리포트

작성일: 2026-06-24 · 대상: `/scratch/x3319a05/Filtered_TDMA`
검증 환경: KISTI Neuron, V100 (`cas_v100nv_8`, sm_70), nvhpc/25.11_cuda12 + hpcx OpenMPI

---

## 0. 요약 (TL;DR)

- **[신규] 주기(periodic) 경계 지원 완료** — heat_cpu + heat_gpu, 3 백엔드(pascal /
  filtered / filtered_v2) 전부. 입력 `periodic = 1`(전 3방향) 또는 `periodic_x/y/z = 1`.
  V100에서 2차 정확도 검증, CPU↔GPU bit-identical(pascal).
- **[신규] filtered CUDA cyclic 포팅** — GPU에 없던 `solve_cycl_filtered_v1/v2`를
  `FilteredTDMACUDA`에 추가(이전엔 PaScaL GPU만 cyclic 보유).
- **[버그 수정] GPU ghost 교환 np==1 누락** — `ghostcellUpdateDevice`가 `nprocs>1`로
  가드돼 단일-rank 주기에서 wrap이 안 됨 → 저차 오차(order ~0.5). 이웃 존재(`west/east
  != MPI_PROC_NULL`) 기준으로 수정.
- **order-of-accuracy 현황** — 세 백엔드 모두 공간 2차(order→2.0). 현재 HEAD의 Dirichlet
  L2(N=64)는 **CPU·GPU 공통 0.000124416** (README의 4.21191e-5는 stale, §6).
- **스케일링 분석** — PaScaL vs Filtered 전체 시간차 ~0–3%. 차이는 분해 방향 `solve_z`
  에서만(rhs가 ~68% 지배), 이득은 **프로세스 수에 비례**(격자 크기엔 무관). cutoff `J`는
  격자에 로그적으로만 증가하는 작은 상수.

## 1. [신규] 주기(periodic) 경계 지원

### 1.1 무엇을
heat ADI 솔버(heat_cpu, heat_gpu)가 방향별로 Dirichlet 벽 또는 주기 경계를 선택할 수
있게 됨. `periodic = 1`이면 x·y·z 전부 주기. manufactured solution
`sin(πx)sin(πy)sin(πz)·exp(−3π²t) + cos(πx)cos(πy)cos(πz)` 은 주기 2 = 도메인 길이라
**주기 문제의 정확해이기도** 하여 동일한 order 테스트로 검증.

### 1.2 왜 대부분 "배선"이었나
- cyclic 솔버가 이미 존재: CPU `solve_cycl_filtered_v1/v2`, PaScaL CPU/GPU `solve_cyclic`.
- cyclic 멀티-rank 본문은 non-cyclic과 **동일** — wrap은 오로지 periodic 커뮤니케이터
  (rank0의 왼쪽 이웃 = 마지막 rank)에서 나옴.
- cell-centered 격자(ddx=lx/(nx-1), 내부 nx-1셀)가 이미 주기-정합(seam 간격도 ddx).

### 1.3 변경 사항 (방향별 동일 패턴)
| 위치 | 변경 |
|---|---|
| `global.{hpp,cpp}` | `std::array<bool,3> periodic` + `periodic`/`periodic_x/y/z` 파싱 |
| `main.cpp` | `topo.init(..., {periodic[0],periodic[1],periodic[2]})` |
| `mpi_subdomain.cpp` `indices()` | 주기 방향은 Dirichlet 인덱스 플래그 미설정 → `boundary()`·ghost Dirichlet fill·`build_lhs` Dirichlet 보정이 자동 skip(전부 플래그 게이트); ghost는 wrap이 채움 |
| `tdma_backend{,_gpu}.{hpp,cpp/cu}` | `solve_cyclic()` 디스패치 추가 (pascal→solve_cyclic, filtered→solve_cycl_v1, filtered_v2→solve_cycl_v2) |
| `solve_theta.{cpp,cu}` | 방향별 `periodic[d] ? solve_cyclic : solve` 분기 |

### 1.4 GPU 전용 작업
1. **filtered CUDA cyclic 포팅** (`libs/filtered_tdma/filtered_tdma_cuda.{hpp,cu}`):
   `solve_cycl_filtered_v1/v2` 추가. np==1 → `tdma_cyclic_many_cuda`(Sherman-Morrison,
   lazy-alloc `d_E_` [n_row×n_sys] workspace 필요); np>1 → non-cyclic과 공유하는
   private `v1_multirank/v2_multirank` 헬퍼(중복 제거 리팩터).
2. **ghost 교환 버그 수정** (`apps/heat_gpu/ghostcell_cuda.cu`): 방향별 halo를
   `if (cx.nprocs > 1)`로 가드 → np==1 방향은 교환 자체를 안 함. Dirichlet은 ghost
   커널이 채우니 무해했지만, **주기 + np==1이면 wrap이 안 일어나** explicit RHS가 잘못된
   ghost를 써서 order가 ~0.5로 붕괴. 수정:
   `if (cx.west_rank != MPI_PROC_NULL || cx.east_rank != MPI_PROC_NULL)`
   (주기 np==1은 west==east==self → self-send가 ghost를 wrap). CPU의 DDT 기반
   `ghostcellUpdate`엔 이 가드가 없어 처음부터 정상이었음.

### 1.5 검증 (V100, order 테스트, N=64/128/256)
| 구성 | pascal | filtered | filtered_v2 | order |
|---|---|---|---|---|
| CPU np=1 / 2×2×2 | 4.3936e-5 → 2.766e-6 | 동일 | 동일 | ~1.99 |
| GPU np=1 (전 주기) | **CPU와 bit-identical** | 동일 | 동일 | ~1.99 |
| GPU np=2 (z분해) | 1GPU와 동일 | 4.39367e-5(~1e-5 내) | 4.39369e-5 | ~1.99 |

- np=1 ↔ 멀티-rank **np 불변**(랭크 간 wrap 정확). 포팅한 filtered CUDA cyclic도
  2GPU에서 정상 2차.
- Dirichlet 회귀(periodic=0): 0.000124416 — 변화 없음(주기 코드가 Dirichlet 경로를
  건드리지 않음, git stash로 HEAD 동일값 확인).

### 1.6 실행 주의 (V100 멀티-GPU)
- **빌드**: `gcc/15.2.0 + mpi/openmpi-4.1.8 + nvhpc/25.11_cuda12` (mpicxx는 nvhpc hpcx로 해석).
- **런타임**: **nvhpc만 로드** (openmpi 추가 로드 시 hpcx mpirun이 가려져 멀티-GPU
  CUDA-aware MPI 실패).
- `srun -n1` 스텝 안에서 `mpirun -np N`은 `--oversubscribe` 필요(아니면 "not enough slots").
- 테스트 드라이버: `apps/heat_gpu/run_periodic_test.sh`.

## 2. order-of-accuracy 현황

- 세 백엔드(pascal/filtered/filtered_v2) × Dirichlet/주기 모두 **공간 2차**
  (E(h)/E(h/2) → 4.0, order → 2.0). filtered의 절단 오차는 round-off 수준(자릿수 일치).
- `option = order`: `dt = ρ/(1−2ρ)·2dx²` (ρ=0.25 ⇒ dt=dx²), `Tmax` 고정 →
  N=64/128/256/512에서 Nt=2/8/32/128. 모든 N에서 T_final 동일 → 순수 공간 order 측정.
- **현재 HEAD Dirichlet L2(N=64) = 0.000124416** (CPU·GPU 공통). 이는 ghost-cell 경계
  재작성(전 영역 ρ<1/2) 이후의 값으로, README/이전 표의 `4.21191e-5`(반-셀 경계 시절,
  현재 삭제된 입력 기준)와 다름.

산출물: `apps/heat_gpu/results/order_v100/`(GPU), `Report/order_*.out`(과거 sbatch 로그).

## 3. CPU 스케일링 분석 (PaScaL vs Filtered)

데이터: `apps/heat_cpu/results/scaling_cpu/` (108 CSV, strong/weak/refine × np2/4/8 ×
2 백엔드 × 2 rho). 분해는 항상 (1,1,np) z-slab. 지표 = 스텝당 wall(이벤트별 critical path 합).

### 3.1 핵심 결론
- **전체 시간차 ~0–3%.** 한 스텝 비용: rhs ~68%(백엔드 무관) · solve_x+y ~17–20%(nprocs=1
  → 두 백엔드 동일) · **solve_z ~13–15%(유일한 차이)** · comm ~0.05%.
- Filtered는 solve_z에서 ~20–30% 빠르지만 전체로 희석. **이득은 분해 방향 프로세스 수에
  비례**(solve_z 가속 np=8에서 PaScaL ~2.5× vs Filtered ~3.1×), **격자 크기엔 무관**.
- 그래서 refine(np=2 고정)에서는 차이가 거의 안 보임 — filtered가 가장 불리한 구간.

### 3.2 왜 — cutoff `J`
`solve_filtered_v2`(filtered_tdma.cpp:311)는 비분해 방향(nprocs=1)에서 일반 Thomas로
폴백 → PaScaL과 byte-동일. 분해 방향에서만 J-절단 작동. `J = ⌊log(eps_/B)/log(q)⌋+1`,
`q=rho/λ₊`, `eps_=dt/N²`. J는 작은 상수이고 격자에 **로그적으로만** 증가 → 절약이
안 커짐. (요청 분석 결과로 `if(has_left/has_right)` 분기는 제거 불필요 — 루프 밖 상수
분기 + MPI_PROC_NULL 안전장치, 비용 무시.)

### 3.3 산출물
- 플롯: `apps/heat_cpu/results/cpu_image/` (strong/weak/refine + refine_all + metrics.csv)
- z-solve 단독: `apps/heat_cpu/results/cpu_image/z_only/` (+ refine_zratio = PaScaL/Filtered
  solve_z 비, refine에서 ~1.2배 평평)
- 스크립트: `apps/heat_cpu/results/plot_scaling_cpu.py`, `plot_scaling_z.py`

## 4. GPU 스케일링 출력 경로 분리

`scaling_gpu/`에 V100/A100 결과가 같은 파일명으로 덮어쓰던 문제 → **하드웨어명 하위
폴더로 자동 분리**. run/sbatch 스케일링 스크립트(`apps/heat_gpu/scripts/`)에
`nvidia-smi`로 GPU명을 감지해 `scaling_gpu/<v100|a100|h100|…>`에 기록하는 블록 추가
(감지 실패 시 스크립트별 fallback). `sbatch_scaling_a100.sh`의 `TIMING_CSV` 누락 버그도
함께 수정. 플롯(`plot_scaling_gpu.py`)은 GPU 인자로 폴더 선택. 기존 108개 루트 CSV는
backup으로 보존.

## 7. 빌드 / 실행 빠른 참조

```bash
# CPU
module load gcc/15.2.0 mpi/openmpi-4.1.8 fftw3/3.3.10
make heat                       # build/bin/heat.out
mpirun -np 8 build/bin/heat.out <input.txt>

# GPU (V100=70 / A100=80 / H100=90)
module load gcc/15.2.0 mpi/openmpi-4.1.8 nvhpc/25.11_cuda12
USE_CUDA=1 CUDA_ARCH=70 make heat_gpu      # 헤더만 바뀌면 make clean 먼저(.cu는 -MMD 없음)
# 실행은 nvhpc만 로드 + srun 안 mpirun엔 --oversubscribe

# 입력 핵심 키
#   option = order|strong ,  tdma_backend = pascal|filtered|filtered_v2
#   periodic = 1  (전 3방향 주기; 또는 periodic_x/y/z = 1)
```
